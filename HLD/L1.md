## 1. HLD vs LLD

```
          System Design
          /            \
        HLD             LLD
  (Architecture)    (Code Level)
```

| Feature | HLD (High Level Design) | LLD (Low Level Design) |
|---|---|---|
| Focus | Architecture / Big Picture | Coding patterns, design principles |
| Examples | Load balancing, Databases, Caching | Design patterns (SOLID, OOPS) |
| Code Level | No actual code | Clean code, loosely coupled |
| Scope | Large-scale system decisions | Class/function level decisions |

**HLD covers:**
- Architecture decisions
- Load balancing
- Caching
- Queues
- Internal architecture design

**LLD covers:**
- Coding (OOP, SOLID, design principles)
- Design patterns
- Clean Code
- Loosely coupled code

## 3. Client-Server Model

### HLD → Architecture → Application

```
Website / Mobile Application

[Web Browser]  ──── HTTP/HTTPS request ────►  [Server]
[Mobile App]   ◄─── response ──────────────   (processing)

         CLIENT                    SERVER
```

### Request-Response Model (Client-Server Model)

```
[Web Browser]  ──── HTTP/HTTPS request ────►  [Server 1] (RAM: 32GB)
[Mobile App]   ◄─── response ────────────────
                                               [Server 2]
        1000 users
                     Distributed System        [Server 3]
```

**Key Concept:**
> Ek server sirf itne hi users handle kar sakta hai. Jab users badhte hain → **Distributed System** chahiye!

## 4. Distributed Systems & Scaling

```
            [Client]
                |
           Internet
                |
         [32GB] [32GB] [32GB]
              SERVERS
```

### Two types of Scaling:

| Type | Meaning | How? |
|---|---|---|
| ⬆️ **Vertical Scaling** | Ek hi server ko powerful banana | RAM/CPU upgrade karo (32GB → 64GB) |
| ↔️ **Horizontal Scaling** | Zyada servers add karna | Naye servers add karo (preferred ✅) |

> 💡 **Real world mein Horizontal Scaling preferred hai** — because vertical scaling has hardware limits!

## 7. Application Layer Protocols

### A) FTP / SFTP
```
[Client] ──── FTP/SFTP ────► [Server]
```
- **FTP** = File Transfer Protocol
- **SFTP** = Secured FTP (encrypted)
- Use: Files transfer karne ke liye

### B) SMTP (Simple Mail Transfer Protocol)
- Use: **Email bhejne ke liye**
- Related: POP3/IMAP (Mail receive karne ke liye)

### C) HTTP (HyperText Transfer Protocol)
```
        [Server]
       /   |
      req  |
     /     |
[Client1] [Client2]
```
- **HTTP** = Request-Response based
- Ek client request bhejta hai, server response deta hai
- **HTTP (OLD):** Ek taraf (one-way) communication

### D) WebSockets 🔥
```
[Client1] ──────────── [Server]
                    (always connected)
[Client2] ──────────── /
```
- **Bidirectional** (dono taraf data flow)
- Connection **persistent** rehta hai
- Use: **Chat apps, Live notifications, Real-time apps**
- Example: WhatsApp, Stock ticker, Online gaming

### E) WebRTC (Peer-to-Peer)
```
[Client1] ←────────────────► [Client2]
              (peer to peer)
                 [Server]
              (NO need of server!)
```
- **Direct peer-to-peer** communication
- Server ki zaroorat nahi (ya minimal)
- Use: **Video calls (Google Meet, Zoom)**

## 8. TCP vs UDP

### TCP — Transmission Control Protocol
> **"Reliable (SLOW)"**

```
msg
packets: p1, p2, p3

Client  ──── p1 ────►  Server
Client  ◄─── ack ────  Server   ✅ p1 received!
Client  ──── p2 ────►  Server
Client  ◄─── ack ────  Server   ✅ p2 received!
```

**How it works:**
- Har packet bhejo → acknowledgement (ack) ka wait karo
- Agar packet lost? → **Dobara bhejo**
- Isliye **Reliable** hai ✅
- Par **Slow** hai ❌ (ack ka wait karna padta hai)

**Use Cases:** Email, File transfer, Banking, Web pages

---

### UDP — User Datagram Protocol
> **"Non-Reliable (FAST)"**

```
Client  ──── p1 ────►  Server
Client  ──── p2 ────►  (LOST ❌ — koi tension nahi!)
Client  ──── p3 ────►  Server
```

**How it works:**
- Packets blast karo, **ack ka wait nahi**
- Packet drop? → **Chhod do, aage badho!**
- Isliye **Fast** hai ✅
- Par **Unreliable** hai ❌

**Use Cases:** YouTube streaming, Online gaming, Video calls, Live sports

---

### TCP vs UDP — Comparison Table

| Feature | TCP | UDP |
|---|---|---|
| Full Name | Transmission Control Protocol | User Datagram Protocol |
| Speed | 🐢 Slow | ⚡ Fast |
| Reliability | ✅ Reliable | ❌ Unreliable |
| Acknowledgement | ✅ Haan (ACK) | ❌ Nahi |
| Connection | Connection-oriented | Connectionless |
| Packet Loss | Re-transmits | Drops silently |
| Use Case | Banking, Email, Web | Video, Gaming, Calls |

> 🧠 **Trick to remember:**
> - **TCP = Registered Post** (confirm karta hai delivery)
> - **UDP = Pamphlet/Flyer** (phenk diya, dekha na dekha!)

## 9. DNS — Domain Name System

### Problem:
- Computer sirf **IP addresses** samajhta hai → `120.23.23.2`
- Insaan **names** yaad rakhta hai → `youtube.com`
- **DNS** in dono ke beech bridge ka kaam karta hai!

### DNS = Internet ka Phone Book 📖

| URL | IP Address |
|---|---|
| https://www.youtube.com | 120.23.23.2 |
| https://www.instagram.com | 120.23.24.4 |

### How DNS Works — Step by Step:

```
Step 1: Browser mein type kiya → youtube.com
Step 2: DNS Server se pucha → "youtube.com ka IP kya hai?"
Step 3: DNS ne reply kiya → IP: 120.23.23.2
Step 4: Browser us IP pe request bheja
Step 5: YouTube ke server (s1 / s2) ne respond kiya
Step 6: YouTube screen pe aa gaya! 🎉
```

### DNS Flow Diagram:
```
[youtube.com]
      |
   [Client]
      |
      ▼
    [DNS]  ────────────────────► [URL → IP Table]
  (Domain                         youtube.com = 120.23.23.2
  Name System)
      |
      | IP: 120.23.23.2
      ▼
  [Web Browser]
      |
      ▼
  [YouTube Servers]
    ┌────┐
    │ s1 │
    ├────┤
    │ s2 │
    └────┘
```

> 💡 **IP = Internet Protocol Address** — Har server ka unique address hota hai internet pe

## 🔁 Quick Revision Cheatsheet

```
HLD = Architecture (Big Picture)
LLD = Code Level (Design Patterns)

Client → Request → Server → Response → Client
         (HTTP/HTTPS)

Scaling:
  Vertical   = Single server ko upgrade karo
  Horizontal = Zyada servers add karo ✅ (preferred)

OSI Layers (remember: All People Seem To Need Data Processing)
  Application → Transport → Network

Protocols:
  FTP/SFTP   = File transfer
  SMTP       = Email
  HTTP       = Web (Request-Response)
  WebSocket  = Real-time bidirectional (Chat apps)
  WebRTC     = Peer-to-peer (Video calls)
  TCP        = Reliable, Slow (Banking, Email)
  UDP        = Fast, Unreliable (Video, Gaming)

DNS = URL → IP address converter (Internet ka Phone Book)
```
